Receptionist Module

Based on analysis of the existing WinMD PolyClinic system. This module covers everything a receptionist does from the moment a patient walks in — registration, appointment booking, service selection, billing, payment collection, and patient history lookup. All workflows observed from the existing system have been mapped, improved, and broken into dev tasks.

What the existing system does — observed from existing system Patient registration (Add New Patient form with 20+ fields including Aadhaar), appointment management with department and doctor filters, service appointment creation (Lab, X-Ray, USG, Other Service), billing with line items and discount, payment collection (Cash/Cheque with receipt generation), patient history view (OPD + IPD with all service dates), lab report entry with abnormal value highlighting, WhatsApp receipt sharing.
What we are improving over the existing system Web-based (not desktop), role-scoped so receptionist sees ONLY what they need, Aadhaar and contact details encrypted at rest, no clinical data visible (diagnosis/lab results hidden), token/queue number system, WhatsApp notification via API instead of manual sharing, consent captured digitally at registration, audit log on every action.

Module 2 · Complete workflow

Receptionist Workflows

Six primary workflows the receptionist performs daily. Click each step to expand implementation detail. These are derived directly from the existing system plus improvements for the web platform.

Workflow 1 — New patient walk-in
Step 1
Search if patient already exists
GET /api/patients?search= · phone or name
  • Search by phone number (primary) or name
  • Phone lookup uses hash index — fast without decrypting all records
  • If found → open existing patient profile → proceed to appointment
  • If not found → proceed to registration
  • Results show: name (masked), patient ID, last visit date only
Step 2
Register new patient
POST /api/patients · consent captured
  • Required: first_name, last_name, phone, gender, date_of_birth
  • Optional: Aadhaar, address, blood group, referred_by, department
  • Consent checkbox must be checked before save (DPDP compliance)
  • Patient ID auto-generated (sequential + UUID)
  • Checklist: admin's "patient_added" checklist item updates automatically
  • Audit log: PATIENT_REGISTERED with receptionist user_id
Step 3
Book appointment
POST /api/appointments · assigns token
  • Select department → filter available doctors
  • Select doctor → view available slots for today/date
  • Select time slot → book appointment
  • Token number auto-assigned (sequential per day per doctor)
  • Appointment status: scheduled
  • Optional: enter referring doctor name
Step 4
Patient arrives — check in
PATCH /api/appointments/:id/status → checked_in
  • Find appointment in today's queue
  • Mark as checked_in — triggers token display
  • Appointment status: scheduled → checked_in
  • Doctor's queue updates in real time
  • Check-in time recorded for analytics
Step 5
Add services to visit
POST /api/visits/:id/services · Lab/X-Ray/USG/OPD
  • Select service type: Lab, X-Ray, USG, OPD Consultation, Other
  • Select specific items from price list (e.g. ESR ₹100, CRP ₹350)
  • Multiple items can be added in one visit
  • Referring doctor selected per service
  • Auto-calculates total from price list
  • Creates service orders — visible to Lab Tech / Pharmacist in their module
Step 6
Generate invoice + collect payment
POST /api/billing/invoices + POST /api/billing/payments
  • Invoice auto-generated from services added
  • Apply discount if applicable (requires reason)
  • Collect payment: Cash / UPI / Card / Cheque
  • For cheque: capture cheque no, date, bank name
  • Receipt auto-generated with sequential receipt number
  • Send receipt via WhatsApp API or print
  • Due balance tracked if partial payment
Workflow 2 — Returning patient
Search by phone → existing profile opens → book appointment or add service to existing visit → generate invoice → collect payment → send receipt. Same flow as new patient from Step 3 onwards.
Workflow 3 — View patient history (receptionist view)
Receptionist sees only: Patient name, contact, past appointment dates, invoice amounts, payment status, pending dues. They do NOT see: diagnosis, lab results, prescriptions, clinical notes. Separate API endpoint (/api/patients/:id/basic) enforces this at the data layer — not just the UI.
Workflow 4 — Collect due payment
Patient lookup → view billing history → find invoice with due amount → POST /api/billing/payments against that invoice → receipt generated. The existing system shows this clearly — patient may pay a previous balance separately (seen in View All screen with due amounts).
Workflow 5 — Cancel / reschedule appointment
Find appointment in queue → PATCH status to cancelled with reason → system logs action with user_id → patient SMS notification triggered. Reschedule = cancel + rebook. Both actions logged in audit trail. Receptionist cannot delete — only cancel with reason.
Workflow 6 — Print / WhatsApp receipt or bill
From billing screen → select invoice → Print (generates PDF) or WhatsApp (sends via WhatsApp API to patient's registered number). Combine bill print merges multiple services into single invoice — matches existing "Combine Bill Print" feature seen in existing system. Reprint available from payment history.
